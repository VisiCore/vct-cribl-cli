# Architecture

Developer guide to the cribl-cli codebase.

## Project structure

```
cribl_cli/
  __init__.py              # Package marker
  __main__.py              # Entry point for `python -m cribl_cli`
  cli.py                   # Click command tree, CriblCLI group with auth init

  api/
    client.py              # httpx client, AuthTransport, DryRunTransport
    endpoint_factory.py    # Generic CRUD Endpoints class, scope URL builders
    endpoints/             # Hand-written endpoint modules (18 files)
      sources.py, destinations.py, pipelines.py, routes.py,
      search.py, notebooks.py, edge.py, edge_nodes.py,
      workers.py, version.py, system.py, jobs.py, metrics.py,
      kms.py, preview.py, logger.py, profiler.py

  auth/
    oauth.py               # Cloud (OAuth2) and local auth, token caching

  commands/
    command_factory.py      # Generates Click CRUD subcommands from registry
    registry.py             # Declarative list of 49 factory-generated commands
    config_cmd.py           # config set/show/use (no auth required)
    workers.py, sources.py, destinations.py, pipelines.py,
    routes.py, search.py, notebooks.py, edge.py, jobs.py,
    version.py, system.py, metrics.py, kms.py, preview.py,
    logger.py, profiler.py, health.py, overview.py

  config/
    loader.py               # Load/save ~/.criblrc, env var + CLI flag merging
    types.py                 # ProfileConfig, CriblConfig, CriblRc dataclasses

  output/
    formatter.py             # JSON (default) and table (tabulate) formatting

  utils/
    errors.py                # CriblApiError, AuthenticationError, DryRunAbort, handle_error
    group_resolver.py        # Auto-resolve worker group when not specified
    pagination.py            # Pagination helpers
    unwrap.py                # Unwrap single-item API responses
    validation.py            # JSON argument parsing

tests/
  unit/                      # 71 unit tests (pytest + responses)
  integration/               # Integration tests (gated behind env var)
```

## Key patterns

### Command registration

Commands are registered in `cli.py:_register_commands()`:

1. **Hand-written commands** (19 groups) — imported and added via `cli.add_command(group)`
2. **Factory-generated commands** (49 groups) — defined declaratively in `registry.py`, generated by `command_factory.py:register_crud_command()`

### Two-tier endpoint design

**Factory endpoints** (`endpoint_factory.py`) — generic `Endpoints` class with `list/get/create/update/delete` methods. Used by factory-generated commands for standard CRUD resources.

**Hand-written endpoints** (`api/endpoints/*.py`) — for resources that need special logic:
- Routes: insert-before-catch-all, edge vs stream format detection
- Search: NDJSON result parsing, job polling
- Notebooks: section/info block structure, JWT identity
- Edge nodes: file inspection, log scanning, metrics aggregation
- Workers: deploy with compound version parameter

### HTTP client transport stack

```
Request
  -> AuthTransport       (injects Bearer token)
    -> DryRunTransport   (logs and aborts if --dry-run, optional)
      -> httpx.HTTPTransport (sends request)
```

The `CriblCLI` group class in `cli.py` initializes the client before any non-config command runs. Config commands skip auth entirely.

### API endpoint scopes

Four URL patterns based on endpoint scope:

| Scope | URL pattern | Default group |
|---|---|---|
| `group` | `/api/v1/m/{group}/{path}` | Auto-resolved from API |
| `global` | `/api/v1/{path}` | N/A |
| `search` | `/api/v1/m/{group}/search/{path}` | `default_search` |
| `lake` | `/api/v1/products/lake/lakes/{lakeId}/{path}` | Required |

### Config priority chain

```
CLI flags (--base-url, --client-id, etc.)
  > Environment variables (CRIBL_BASE_URL, CRIBL_CLIENT_ID, etc.)
    > Active profile in ~/.criblrc
      > "default" profile in ~/.criblrc
```

### Error handling

All commands wrap their logic in `try/except Exception as e: handle_error(e)`.

`handle_error()` outputs structured JSON to stderr:
- `DryRunAbort` — exits 0 (success)
- `CriblApiError` — `{error, status, detail}` to stderr, exits 1
- `AuthenticationError` — `{error, type}` to stderr, exits 1
- Other — `{error}` to stderr, exits 1

### Route table safety

Routes use a fetch-modify-patch pattern:
1. Fetch entire route table
2. Insert new route before the catch-all (last `final: true` route)
3. PATCH the full table back

Edge groups have a different route table structure (`items[0].routes` vs `items`) and require PATCH to `/routes/{tableId}` instead of `/routes`.

### Update merge pattern

Update operations for sources, destinations, and pipelines:
1. GET the existing resource
2. Strip server-computed fields (`status`, `notifications`, etc.)
3. Deep-merge user-provided JSON
4. PATCH the result

This prevents accidental overwrite of computed fields.

## Development

```bash
# Setup
python3 -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"

# Run tests
pytest                                    # all unit tests
pytest tests/unit/test_routes.py          # specific file
pytest -m "not integration"               # skip integration tests (default)
CRIBL_INTEGRATION_TEST=true pytest -m integration  # run integration tests

# Run the CLI in development
python -m cribl_cli workers list
# or use the bin wrapper:
bin/cribl workers list
```

### Adding a new standard CRUD command

Add an entry to `registry.py`:

```python
CommandRegistration("my-resource", EndpointConfig("group", "system/my-resource")),
```

This auto-generates `cribl my-resource list|get|create|update|delete`.

### Adding a hand-written command

1. Create `api/endpoints/my_thing.py` with endpoint functions
2. Create `commands/my_thing.py` with Click group and subcommands
3. Import and register in `cli.py:_register_commands()`

### Testing

- Unit tests use `pytest` with `responses` for HTTP mocking and `pytest-mock` for general mocking
- Config tests mock `~/.criblrc` to avoid touching real config
- Integration tests require `CRIBL_INTEGRATION_TEST=true` and a live Cribl instance
